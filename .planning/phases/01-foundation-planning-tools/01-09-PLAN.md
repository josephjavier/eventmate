<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| invitee link → accept route | Token authorizes the grant; must be single-use |
| Server Action → Resend | API key server-only; email is transactional |
| RSVP respond → owner email lookup | Service client reads owner email narrowly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-15 | Spoofing | Co-planner invitation forgery / replay | mitigate | `nanoid(20)` unguessable token; accept route rejects already-accepted tokens (one-time-use) and requires the invitee to be authenticated |
| T-1-16 | Elevation of Privilege | Unauthorized event access via accept | mitigate | Access flows only through `user_can_access_event()` (status='accepted' AND user_id=auth.uid()); pending invites grant nothing |
| T-1-17 | Information Disclosure | Resend API key exposure | mitigate | `RESEND_API_KEY` server-only (no NEXT_PUBLIC_); used only in Server Actions / Route Handlers |
| T-1-18 | Denial of Service | RSVP fails if email send fails | mitigate | D-08 notification wrapped in try/catch — best-effort, never blocks the guest RSVP write |
</threat_model>

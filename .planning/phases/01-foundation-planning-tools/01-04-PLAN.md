<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous visitor → signup | Untrusted email/password/consent input crosses here |
| client → Server Actions | All event writes must re-verify session via DAL |
| Vercel env → browser | Only NEXT_PUBLIC_* may reach the client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-05 | Spoofing | Forged session on protected routes | mitigate | `assertRole('client')` in (client) layout + `verifySession()` first line of every Server Action (getUser-based) |
| T-1-08 | Tampering | Unvalidated signup/event input | mitigate | Zod v4 schemas validate client-side AND the Server Action parses/validates server-side before insert |
| T-1-DPA | Privacy violation (RA 10173) | Consent capture | mitigate | Required consent checkbox; `consent_given_at` written on signup before reaching the app |
| T-1-06 | Elevation of Privilege | Service-role key in Vercel | mitigate | SUPABASE_SERVICE_ROLE_KEY set without NEXT_PUBLIC_ prefix; used only server-side in keepalive route |
| T-1-09 | Denial of Service | Supabase auto-pause | accept/mitigate | Keep-alive cron every 3 days keeps the free-tier project warm (Pitfall 7) |
</threat_model>

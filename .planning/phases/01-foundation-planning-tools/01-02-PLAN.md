<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated client → Postgres | Every row read/write is mediated by RLS; the only hard authorization boundary |
| co-planner (second account) → event rows | Access granted only via an accepted invitation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-01 | Information Disclosure | Cross-client data access | mitigate | RLS enabled in the same migration as every CREATE TABLE; event-scoped tables gated on `user_can_access_event()` (Task 1) |
| T-1-02 | Spoofing | Co-planner access grant | mitigate | Access requires `event_invitations.status = 'accepted'` AND `user_id = auth.uid()` inside the helper; pending invites grant nothing |
| T-1-03 | Tampering | Money columns as float | mitigate | Acceptance criterion forbids `numeric`/`decimal`; all monetary columns `INTEGER` centavos |
| T-1-04 | Information Disclosure | Storage objects | mitigate | All three buckets `public = false`; storage.objects policies gate on `user_can_access_event(event_id-folder)` |
| T-1-DPA | Privacy violation (RA 10173) | profiles | mitigate | `consent_given_at timestamptz` column captured at signup (Wave 4) |
</threat_model>
